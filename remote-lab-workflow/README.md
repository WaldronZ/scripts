# remote-lab-workflow

一套「本地优先 + 远程实验」工作流的 `CLAUDE.md` 模板。把它放进项目作为该项目的
`CLAUDE.md`（按项目改配置），Claude Code 每个 session 开始会先读它，然后照章办事。

## 有什么内容

核心仪式（任何项目通用）：

- **循环**——session 开局先读 `docs/state.md` 全文与 `docs/progress.md` 尾部（当前状态在这两个文件里，不在 memory 里；发现还有任务在跑，先恢复轮询再开始新修改）；然后：本地修改 → 记日志 → 同步 → 运行 → 拉回结果 → 本地分析。
- **强制实验日志**——每个有意义的步骤 append 进 `docs/progress.md`；append-only；写前 `wc -l` 自检；一轮循环一条日志（就地补全只允许当前条目的 Command/Result 两行）；条目只写引用不写内容；另有可修订的 `docs/state.md` 存放当前工作状态（假设、最佳配置、已知坑）。
- **风险分层 git**——不在脏树上开工（用户在改的进行中修改须询问后提交）；一个逻辑步骤一次提交；低风险直接改，高风险（wrapper、adapter/hook、训练循环、数据管道）copy-then-refactor，好结果打 tag——永远有一个已知良好版本可以回退。
- **Memory 纪律**——memory 只是辅助上下文，不是权威来源；代码和 docs 文件优先；易变状态进文件（历史进 `docs/progress.md`、当前状态进 `docs/state.md`），不进 memory；规则只在错误驱动下增补（同一个错犯两次 / 同样的纠正跨 session 重演）。
- **Guardrails**——不猜路径和配置；宣称完成必须实际运行并验证输出；失败带假设记进日志；不许悄然留下在跑的任务——会话结束时点名交接并记入日志。

远程模块（纯本地项目可整节删除）：

- **配置**——`SERVER` / `REMOTE_DIR` / `RESULTS_REMOTE_DIR` / `EXCLUDES` 内联在文件里；值归用户所有（未经确认不许改写；用户确认过的值可写回，先出示 diff）。
- **同步上行**——带排除项 rsync；未经明确确认永不 `--delete`；首次同步或排除项变化后先 `--dry-run`。
- **远程运行**——`ssh -tt` 执行；长任务前先冒烟测试（脚本没有廉价变体就先加一个）；需要活过 ssh 会话的任务跑在 detached tmux 里（配提交的 `run_<name>.sh` 包装脚本）；任务命名 `YYYYMMDD-HHMM-<slug>`；轮询先密后疏（~30 秒首查，之后 2–10 分钟）；完成判定 = tmux 会话消失 + 日志尾无报错栈 + 有预期产物。
- **拉回分析**——rsync `RESULTS_REMOTE_DIR/`（结果与日志）到本地 `./results/`；在本地用 Claude 分析，不在服务器上分析。

## 有什么用

- **换 session 不用重新解释**——新会话开局读 `docs/state.md`（当前假设、最佳配置、已知坑）+ 日志尾部；还在跑的任务原地恢复轮询。
- **实验历史可复盘**——每个有意义的步骤 append 进 `docs/progress.md`（命令逐字、run ID、关键数字）：「run-07 用的哪组参数」一 grep 就有。
- **改坏了永远退得回**——不在脏树开工、一个逻辑步骤一次提交、高风险 copy-then-refactor、好结果打 tag：永远存在已知良好版本。
- **长任务活得过会话**——训练跑在 detached tmux；会话结束点名交接并记入日志；会话开始恢复轮询；跑完（会话消失 + 日志尾无报错栈 + 有预期产物）才拉结果。
- **「完成」可信**——实际运行并确认输出才算完成，编译通过不算；memory 只是辅助，冲突时文件与代码赢。
- **本地/服务器不漂移**——本地是唯一编辑源，rsync 单向上行，结果拉回本地分析；未经确认永不 `--delete`。
- **记录自己不会烂**——条目只写引用不写内容；阶段结束归档开新文件；规则只在错误重演时才补——文件不无序膨胀。
